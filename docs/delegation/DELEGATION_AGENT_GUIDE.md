# Delegation Agent Guide

## 1. Purpose

Этот документ предназначен для делегируемого исполнителя, который будет брать отдельные сегменты миграции проекта `connect`.

Цель исполнителя:
- выполнять сегмент так, как если бы его делал основной архитектор проекта
- не ломать текущий runtime
- не уходить от согласованной архитектурной стратегии
- двигать проект по заранее зафиксированному плану, а не по импровизации

Этот документ не заменяет архитектурный план. Он определяет, **как именно** нужно работать внутри каждого сегмента.

## 2. Source of Truth

Перед началом любой работы исполнитель обязан считать следующими документами:

1. [PLATFORM_MIGRATION_PLAN.md](../roadmap/PLATFORM_MIGRATION_PLAN.md)
2. [ARCHITECTURE.md](../roadmap/ARCHITECTURE.md)
3. [BOUNDARIES.md](../roadmap/BOUNDARIES.md)
4. [STAGE_STATUS.md](../roadmap/STAGE_STATUS.md)
5. текущий wave-документ, если он есть
6. текущий `SEGMENT_BRIEF`

Если между локальными идеями исполнителя и этими документами есть конфликт, приоритет у документов.

## 3. Strategic Context

Проект движется к следующей модели:
- одна repo `connect`
- monorepo-структура внутри неё
- `desktop-first` архитектура
- backend в `Nest`
- shared packages для `app-core`, `sdk`, `ui`, `config`
- постепенный уход от:
  - `Clerk`
  - `UploadThing`
  - `LiveKit`

Ключевой принцип:
- не делать большой rewrite
- делать управляемую миграцию через безопасные, контролируемые сегменты

## 4. Non-Negotiable Rules

Исполнитель обязан соблюдать следующие правила.

### 4.1. Do Not Break Current Runtime

Если сегмент не требует изменения runtime entrypoints, их нельзя трогать.

Не трогать без прямой необходимости:
- `src/app/*`
- `src/pages/api/*`
- `src/middleware.ts`
- `electron/*`
- auth/bootstrap/runtime wiring

### 4.2. Preserve Production Behavior

Если задача про структуру, адаптеры, границы или вынос reusable-кода, поведение приложения не должно меняться.

### 4.3. Prefer the Smallest Safe Change

Не делать широкие рефакторы “раз уж мы тут”.

Нужно делать только то, что нужно для конкретного сегмента.

### 4.4. Inspect Before Changing

До любого изменения:
- прочитать релевантные файлы
- понять текущий flow
- проверить соседние модули
- понять существующие зависимости

Нельзя менять код на основе догадок.

### 4.5. Follow the Target Architecture

Исполнитель не должен изобретать новую архитектуру.

Он должен двигать проект к уже согласованной схеме:
- `apps/web`
- `apps/desktop`
- `apps/api`
- `packages/app-core`
- `packages/sdk`
- `packages/ui`
- `packages/config`
- `infra/*`

### 4.6. No Premature Microservices

Если переносится backend, целевая модель на первом этапе — `Nest modular monolith`, а не набор независимых сервисов.

### 4.7. No Premature Frontend Rewrite

Исполнитель не должен инициировать миграцию `Next -> React` раньше времени.

Сейчас `Next` — это временный shell. Он не является первым объектом демонтажа.

### 4.8. No Hidden Vendor Lock-In

Если сегмент касается:
- auth
- storage
- media
- realtime

то решение должно строиться через внутренние contracts/adapters, а не через прямую размазку SDK по коду.

## 5. Required Working Style

Исполнитель должен работать как аккуратный senior engineer.

Это значит:
- думать про runtime behavior, а не только про типы
- не плодить абстракции без необходимости
- не переносить код механически
- объяснять trade-offs
- держать изменения локальными и проверяемыми

## 6. Segment Execution Protocol

Каждый сегмент должен проходить через один и тот же процесс.

### Step 1. Read the Segment Brief

Исполнитель получает сегмент с описанием:
- цель
- scope
- out of scope
- acceptance criteria
- ограничения

### Step 2. Read Relevant Plan Sections

Исполнитель обязан открыть и прочитать только релевантные разделы:
- `PLATFORM_MIGRATION_PLAN.md`
- `ARCHITECTURE.md`
- `BOUNDARIES.md`
- `STAGE_STATUS.md`
- текущий wave-документ
- текущий `SEGMENT_BRIEF`

Нельзя работать по памяти, если есть риск потерять контекст.

### Step 3. Inspect Current Code

Нужно:
- найти входные точки
- найти зависимости
- найти похожие паттерны
- проверить, какие файлы реально участвуют в сегменте

### Step 4. Define Safe Change Boundary

Перед редактированием исполнитель должен явно понять:
- какие файлы он меняет
- какие файлы не должен трогать
- почему изменение безопасно

### Step 5. Apply the Change

Изменения должны быть:
- минимальными
- логичными
- совместимыми с текущим runtime
- направленными к target architecture

### Step 6. Validate

Минимум:
- typecheck, если применимо
- lint по затронутым файлам, если применимо
- ручная логическая проверка flow

### Step 7. Produce Handoff

По завершении исполнитель обязан оставить короткий handoff:
- что сделано
- что не сделано
- какие файлы изменены
- какие риски остались
- что должен делать следующий сегмент

## 7. Quality Bar

Исполнитель обязан держать следующий стандарт качества:

### 7.1. Good Segment

Хороший сегмент:
- двигает архитектуру вперёд
- не ломает существующий runtime
- уменьшает связанность
- улучшает читаемость границ
- не вносит лишних решений

### 7.2. Bad Segment

Плохой сегмент:
- трогает слишком много файлов
- тянет за собой незапрошенные изменения
- меняет поведение без необходимости
- смешивает migration и feature work
- вносит новую архитектуру поверх уже согласованной

## 8. What the Executor Must Not Do

Исполнитель не должен:
- начинать переписывать проект на другой стек без прямой задачи
- превращать сегмент в “общий cleanup”
- переносить runtime entrypoints в ранних этапах без явной причины
- делать split repo инициативно
- инициировать early microservices
- писать свой WebRTC stack с нуля
- тащить новые зависимости без архитектурной необходимости

## 9. Segment Types

Обычно сегменты будут одного из этих типов:

### 9.1. Structure Segment

Примеры:
- создать package scaffold
- вынести shared types
- ввести adapter interface

Требование:
- без изменения пользовательского поведения

### 9.2. Extraction Segment

Примеры:
- перенести reusable logic в `packages/app-core`
- перенести API client в `packages/sdk`
- перенести UI primitives в `packages/ui`

Требование:
- сохранить старый runtime
- обновить импорты безопасно

### 9.3. Backend Migration Segment

Примеры:
- поднять `Nest` module
- перенести invites
- перенести server/channel/member domain

Требование:
- не делать всё сразу
- переносить вертикальными slice-ами

### 9.4. Dependency Removal Segment

Примеры:
- убрать `Clerk`
- изолировать `UploadThing`
- заменить `LiveKit`

Требование:
- сначала ввести adapter/boundary
- только потом менять реализацию

## 10. Default Acceptance Checklist

Если отдельный сегментный бриф не дал более строгий чеклист, использовать этот.

- изменение локализовано и понятно
- scope не расползся
- текущий runtime не сломан
- целевая архитектура стала ближе
- не появилось лишних новых зависимостей
- есть handoff для следующего шага

## 11. Default Handoff Format

По завершении сегмента исполнитель обязан вернуть результат в таком виде:

### Summary

Коротко:
- что сделано
- зачем

### Changed Files

Плоский список файлов.

### What Changed

По каждому важному файлу:
- что именно изменено
- почему это сделано

### Validation

- что проверено
- что не удалось проверить

### Risks

- остаточные риски
- спорные места

### Next Segment

- что логично делать следующим

## 13. Recommended Delegation Pattern

Если работа идёт по сегментам, правильно делегировать так:

1. один сегмент — один исполнитель
2. у сегмента есть чёткий scope
3. исполнитель работает только в этом scope
4. после завершения оставляет handoff
5. следующий сегмент стартует уже от handoff и master-plan

Такой процесс лучше, чем один длинный чат с размытым контекстом.

## 14. Final Rule

Если исполнитель сомневается между:
- “сделать побольше, раз уж начал”
- “сделать только безопасный архитектурный шаг”

он обязан выбрать второе.

Здесь приоритет:
- контроль
- предсказуемость
- качество миграции

а не скорость ценой хаоса.
